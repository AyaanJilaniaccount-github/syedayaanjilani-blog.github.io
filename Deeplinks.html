What are deeplinks to begin with?

Deeplinks function very similar to how links function in websites.It links different parts of the application bundled code set for proper synergy amongst that.
A login part of the app can lead to a certain part of the same application if certain requirements are met for example.This is fulfilled by deeplinks.

Thats essentially deeplinks really.It can be app to app or web to app and vice versa.

It is based on the android intents system.
From https://developer.android.com/training/app-links/create-deeplinks
Deep linking is a general system capability of Android, supported on all versions, on all devices. It takes advantage of Android's Intents system to route deep links to interested apps. Apps that want to handle a specific deep link URI declare a matching Intent filter in their app manifest files.
At runtime, when the user taps a link, Android triggers an intent and attempts to route it to an app. Because multiple apps can declare intent filters that match a given URI, Android takes these actions, in this order, to route the intent:

1.Open the user's default app that can handle the URI, if one was designated.
2.Open the only available app that can handle the URI.
3.Allow the user to select an app from a disambiguation dialog.

Types of Deeplinks

This is well stated in the https://developer.android.com/training/app-links/create-deeplinks site

There are three types of deep links you can support on Android:

1.Custom deep links: These are deep links that use a custom URI scheme (such as example://products/123) to take a user directly to a specific piece of content within an app. They are powerful for internal navigation or links from sources you control, but they are not standard web links and can still trigger the disambiguation dialog if another app registers the same custom scheme.

2.Web links: These are deep links that use the standard http and https schemes. They are more versatile because they are standard URLs, but on Android 12 and higher they will almost always trigger the disambiguation dialog, meaning that they are likely to be handled by the user's web browser by default, rather than being routed to your app.

3.App Links: Available since Android 6.0 (API level 23), these are verified web links. Through a process of website association, you can prove to the Android system that you own the domain. Once verified, the system automatically routes links for that domain directly to your app, skipping the disambiguation dialog entirely. This creates a trusted and seamless experience for your users.

Cool...I know deeplinks now, whats next?

Alright.Now the interesting part...
Deeplinks can be useful in quite alot of critical ways - Useful for the attacker that is in this case.
The core concept is that each part of the application is bundled together and held together with deeplinks.
Applications often follow a sequence from one "page" to the next.Many applications rely on this sequence for its security.
Well what if...this sequence were messed with?
What if one skips the login page directly and goes to the destination page without having to even fill any details?
That would not be good right...
Thats exactly what deeplink exploits try to achieve.It can even make certain surface level security measures useless as those measures were simply bypassed by manually entering the link which the application uses.

Modern android has a lot of guardrails to prevent exactly such exploits.Also it has become standard practice to prevent such things from happening...atleast happening with ease normally.

However, this does not mean that deeplink redirect vulnerabilities are not found even in quite recent days.There are various ways to bypass both the android security guardrails and also the application guardrails as well.
Well lets zoom out of the application links itself.Lets look at what a link is.A link is a connection to another folder that can be accessed by a particular address.
This address can be local or in the world wide web by a specific uniform resource locator(URL) - I wrote that in full form to sort of emphasize what URL really is.It is a resource locator.
Well what if an application's behaviour is manipulated in such a manner that entering a certain input leads not to another application "page" but a self made malicious site?
The user is tricked into going to this malicious site which can be really bad...
This is where the critical or atleast high severity nature of such vulnerabilities arise from.

Well if modern android and even application developers really make measures for exactly such things then why is it such a big deal even?

Well...yea, there is some weight to that.Its not that common to find deeplink exploits as many developers actively prepare against such things.
Agreed.However, there are some significant nuances to be taken into account.
It could be that the application is having measures.Also that the application logic is profoundly sound and lacks significant vulnerabilities, atleast as far as a supposed research is done by a hypothetical malicious hacker.
However, again, lets zoom out a bit.What is an application really in essence?
It is binary instructions.
What executes these binary instructions?
The CPU.
The CPU has its own registers that it uses to perform computation in real time and give the results back.
What stores in stack and heap in memory?
The RAM
When the addresses get corrupted it gives a corrupted instruction to the CPU which in turn processes the instruction in a way that is not intended application flow.
The application sets aside buffers in the memory inside of the RAM with addresses for computation in synergy with the CPU. These buffers can be arbitrarily represented by addresses with a certain value, often expressed in hexadecimal for simplicity.
If...by some way, this buffer could be overloaded with information that the RAM cannot store in that particular memory slot what would happen?
The information would "overflow" like how water overflows when excess is poured in a jug that cannot hold it any longer.
This can corrupt another buffer information potentially messing up the guardrails that were placed, sometimes even at the level of operating system - since it is at the end a software for ease in interacting with hardware.
This sort of exploit is called buffer overflow.With a proper chaining of buffer overflow led memory corruption of the application, one could bypass the very operating system and all of its guardrails.
The CPU executes these corrupted memory instructions which is where the synergy between RAM and CPU is exploited for the attacker to behave like a adminstrator user.

What is the severity really?

Critical.It can even go to Remote Code Execution also known as RCE.Which basically means the name itself - It allows the attacker to inject payloads at the administrator level without being one initially.

How is this relevant to Deeplinks exactly?

It is relevant because when the application logic itself falls apart then deeplinks become the final "hit" to the application's security structure.
This is because the guardrails blocking deeplink redirection attacks have been corrupted to the degree that it cannot recover on time enough to stop this bypass.
These scenarios is what many developers actively work against as it can be immensely damaging if someone malicious manages to get a hold of it.
